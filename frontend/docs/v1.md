# Frontend V1 — Complete Specification

## Stack

| Technology | Purpose |
|------------|---------|
| React 18 + Vite | Application framework and build tool |
| TypeScript | Type safety across components and API contracts |

| React Router | Client-side routing, protected route guards |

## Architecture

### Routing
- Public routes: `/login`, `/register`, `/oauth2/callback`
- Protected routes (require JWT): `/dashboard`, `/settings/amazon`, `/settings/telegram`, `/schedule`
- On 401: clear auth state → redirect to `/login`

### Feature Modules
```
frontend/src/
├── features/
│   ├── auth/           ← login, register, OAuth callback
│   ├── settings/       ← Amazon config, Telegram config
│   ├── schedule/       ← schedule management, activate/deactivate
│   └── dashboard/      ← summary cards, recent posts, category chart
├── components/         ← shared UI components
├── lib/
│   ├── api.ts          ← token-aware axios/fetch wrapper
│   └── auth.ts         ← token storage and session helpers
└── App.tsx             ← router setup and error boundary
```

### State Strategy
- Server state: React Query (all API data — no Redux)
- Local UI state: `useState`/`useReducer` for form inputs only
- Auth token: stored in memory (not localStorage); persisted to sessionStorage as fallback
- Error feedback: centralized toast system; component-level error boundaries for route crashes

### Key Design Decisions
- Business logic stays in backend; frontend handles orchestration and UX only
- All API field names are typed (no `any` on response data)
- Centralized error boundary catches route-level crashes and shows graceful fallback

## Pages and Flows

### 1. Authentication Flow
- User opens `/login` or `/register`
- Submits email/password → `POST /api/auth/login` or `POST /api/auth/register`
- Alternatively clicks "Sign in with Google" → redirected to Google OAuth
- On success: JWT stored, user redirected to `/dashboard` or setup wizard if first time
- On failure: error message shown inline

### 2. Setup Wizard (first-time users)
- Step A: Amazon credentials and affiliate tag → `PUT /api/amazon-config`
- Step B: Telegram bot token and channel ID → `PUT /api/telegram-config` + test button
- Step C: Category and filter preferences (discount %, minimum rating)
- Step D: Schedule time slots and timezone → `PUT /api/schedule` + activate

### 3. Dashboard Flow
- Summary cards: posts today / this week / all time (from `GET /api/dashboard/summary`)
- Recent posts table: title, category, discount, status, platform, timestamp (from `GET /api/dashboard/recent-posts`)
- Category breakdown chart: Recharts bar chart (from `GET /api/dashboard/category-breakdown`)
- Quick links to edit config sections

### 4. Error Flows
- Invalid credentials: inline message on auth form
- Missing/invalid API keys: validation error on config forms
- Unauthorized / token expiry: auto-redirect to login, clear auth state
- Scheduler save failure: toast notification, preserve form state

### UX Rules
- Show empty-state guides for first-time users (no posts yet)
- Use clear action labels: **Save Config**, **Test Connection**, **Activate Schedule**
- Preserve unsaved form state during temporary API errors
- Loading skeletons for async data, not spinners in content areas
- Mobile layout verified on all main pages (320px minimum)

## API Contract

All protected routes require `Authorization: Bearer {accessToken}` header.

### Auth

| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/auth/register` | Register with email + password |
| POST | `/api/auth/login` | Login with email + password |
| POST | `/api/auth/refresh` | Refresh access token |
| POST | `/api/auth/logout` | Revoke token |
| GET | `/oauth2/callback/google` | Google OAuth callback (handled by Spring Security) |

### Config

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/amazon-config` | Get Amazon config (never returns access/secret key) |
| PUT | `/api/amazon-config` | Save Amazon credentials and preferences |
| GET | `/api/telegram-config` | Get Telegram config (never returns bot token) |
| PUT | `/api/telegram-config` | Save Telegram credentials |
| POST | `/api/telegram-config/test` | Send test message to verify connection |

### Schedule

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/schedule` | Get current schedule config and job status |
| PUT | `/api/schedule` | Update schedule preferences |
| POST | `/api/schedule/activate` | Start posting for this user |
| POST | `/api/schedule/deactivate` | Pause posting (config preserved) |
| GET | `/api/schedule/history` | Paginated job run history |

### Dashboard

| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/dashboard/summary` | Posts today / week / all time, success rate |
| GET | `/api/dashboard/recent-posts` | Paginated recent post list |
| GET | `/api/dashboard/category-breakdown` | Posts per category with success counts |

### Response Format

Success:
```json
{
  "success": true,
  "data": { },
  "traceId": "abc-123",
  "timestamp": "2026-02-25T10:30:00Z"
}
```

Failure:
```json
{
  "success": false,
  "data": null,
  "error": {
    "errorCode": "AMAZON_CONFIG_NOT_FOUND",
    "message": "Amazon configuration not set up for this user.",
    "traceId": "abc-123",
    "timestamp": "2026-02-25T10:30:00Z"
  }
}
```

Token behavior:
- Add `Bearer` token to all protected routes
- On `401` response: redirect to `/login` and clear auth state
- Retry idempotent `GET` calls only (not mutations)

## TypeScript Models

```typescript
interface AuthResponse {
  accessToken: string;
  refreshToken: string;
  user: { id: string; email: string; name: string; plan: 'FREE' | 'PRO' | 'AGENCY' };
}

interface AmazonConfigResponse {
  affiliateTag: string;
  categories: string[];
  minDiscountPct: number;
  minRating: number;
  isActive: boolean;
}

interface TelegramConfigResponse {
  channelId: string;
  channelName: string;
  isActive: boolean;
  lastVerifiedAt: string;
}

interface ScheduleResponse {
  postsPerDay: number;
  postingTimes: string[];       // ["09:00", "12:00", ...]
  activeCategories: string[];
  timezone: string;
  isActive: boolean;
  jobStatus: string;
}

interface DashboardSummary {
  postsToday: number;
  postsThisWeek: number;
  postsAllTime: number;
  successRate: number;
  schedulerStatus: string;
}

interface RecentPost {
  productTitle: string;
  category: string;
  discountPct: number;
  status: 'POSTED' | 'FAILED' | 'PENDING';
  platform: string;
  postedAt: string;
}
```

- Loading and error states are visible in all async sections
- Mobile layout verified for all main pages

- No secrets or tokens logged in browser console
- Tokens stored and cleared securely on logout
- Protected routes blocked when not authenticated
