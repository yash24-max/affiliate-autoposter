# Infra V1 — Complete Specification

## Local Development

```bash
docker compose up -d    # Start PostgreSQL + Redis
docker compose down     # Stop services
```

Docker Compose provides PostgreSQL and Redis for local development. The `docker-compose.yml` is located under `infra/`.

## Environment Model

| Environment | Purpose | Target |
|-------------|---------|--------|
| Dev | Local development | Docker Compose |
| Stage | Integration validation | Railway / Render |
| Prod | Live traffic | Railway / Render (V1) |

## Required Environment Variables

All non-dev environments require:

```
DATABASE_URL          PostgreSQL connection string
DB_USERNAME           Database user
DB_PASSWORD           Database password
REDIS_URL             Redis connection string
JWT_SECRET            JWT signing secret
JWT_EXPIRY            Access token TTL (e.g. 900 for 15 minutes)
GOOGLE_CLIENT_ID      Google OAuth client ID
GOOGLE_CLIENT_SECRET  Google OAuth client secret
GOOGLE_CALLBACK_URL   OAuth redirect URL
ENCRYPTION_MASTER_KEY AES-256 master key for encrypting user secrets
APP_ENV               dev | stage | prod
```

## CI/CD Pipeline

**Branch Strategy:**
- `main` → production (manual approval gate required)
- `dev` → staging (auto-deploy on merge)
- Feature branches → PRs to `dev`

**Pipeline Stages (GitHub Actions):**
1. Checkout and dependency restore
2. Build and unit tests (`./mvnw clean package`)
3. Static checks (lint/format if configured)
4. Container image build and tag (versioned tag per commit SHA)
5. Deploy to stage on `dev` merge
6. Deploy to prod on `main` merge (manual approval required)

**Artifacts:**
- Build logs
- Test reports
- Versioned container image tags (for rollback)

**Rollback:** redeploy the prior versioned container image tag.

## Deployment

**V1 Target:** Railway or Render

- Containerized Spring Boot service (each microservice its own container)
- Managed PostgreSQL and Redis where available from the platform
- Health endpoints: `GET /actuator/health` for readiness/liveness checks

**Release Gates:**
- Build and tests pass in CI
- Basic smoke checks pass in stage
- Config/secret validation complete before prod deploy

## Observability

### Signals to Instrument
- Request latency and 5xx error rate (per service)
- Scheduler run outcomes (POSTED vs FAILED per user)
- External API dependency status (Amazon PA API, Telegram Bot API)
- DB connection pool utilization and Redis availability

### Minimum Dashboards
- API health and throughput (request rate, latency p95, error rate)
- Scheduler outcomes (posted vs failed per time window)
- External API dependency status
- DB and Redis availability

### Alert Conditions
- High 5xx error ratio (above threshold for 5 minutes)
- Repeated scheduler failures for the same user
- External API error spikes (Amazon or Telegram returning errors)
- Database connection pool exhaustion

### Logging
- Structured JSON logs
- `traceId` correlation ID on every log line (injected by API Gateway)
- No PII or credentials in any log line

## Backup and Recovery

### Backup Policy
- Daily PostgreSQL backups
- Backup retention: 7 days (stage), 30 days (prod)
- Config backup for scheduler metadata (Quartz JDBC store)

### Recovery Objectives (V1)
- RPO: 24 hours
- RTO: same-day restoration

### Recovery Sequence
1. Restore latest database snapshot
2. Validate schema integrity and critical tables
3. Re-enable scheduler service in controlled mode (single user first)
4. Smoke-check auth flow, config save, and posting flow
5. Re-enable all user schedules

### Disaster Scenarios
- DB corruption → restore from backup, validate, re-enable scheduler
- Region/service outage → redeploy on alternate region or platform
- Secret leakage → rotate secrets immediately (see runbooks)

## Runbooks

### Scheduler Failure Spike
1. Check scheduler metrics and failed job_runs log entries
2. Validate Telegram and Amazon API responses (look for rate limits or auth errors)
3. Pause affected user schedules if failures are user-specific
4. Apply hotfix or rollback if platform-wide

### Deployment Rollback
1. Identify last stable versioned container image tag from CI artifacts
2. Redeploy that tag to the affected environment
3. Verify health endpoints respond correctly
4. Validate auth, config, and posting flow in stage before reapplying to prod

### Secret Rotation
1. Rotate JWT/OAuth/API secrets in CI secret manager or platform env vars
2. Restart services (rolling restart to avoid downtime)
3. Validate auth flow end-to-end
4. Validate Telegram and Amazon API connectivity

### DB Connection Saturation
1. Check pool metrics and slow query logs
2. Temporarily reduce Scheduler Service concurrency (fewer concurrent Quartz threads)
3. Identify and optimize expensive queries
4. Apply index improvements as follow-up

## Security

- All external API secrets (Amazon PA API keys, Telegram bot tokens) encrypted at rest using `ENCRYPTION_MASTER_KEY`
- JWT and OAuth secrets injected via environment variables only — never hardcoded
- Rate-limit auth endpoints and external integration test endpoints (at API Gateway)
- All errors include a `traceId` correlation ID; sanitize error messages before returning to clients
- Auth and config change events logged for audit
- Internal services (`/internal/**`) not exposed through API Gateway — reachable only within Docker network
