# Infra V4 — Delta (Full SaaS Scale)

This document covers only what changes or is added in V4. All prior infra configuration remains. See [v1.md](v1.md) for the full baseline.

---

## Kubernetes (EKS / GKE)

V4 migrates to Kubernetes for container orchestration.

**Horizontal Pod Autoscaling (HPA):**
- API Gateway: scale on request rate
- Scheduler Service: scale on active user count
- Publisher workers: scale on Kafka consumer lag
- Default policy: scale up at 70% CPU/memory, scale down after 5 minutes cooldown

**Namespace isolation:**
- `autoposter-prod` for production workloads
- `autoposter-stage` for staging
- Network policies enforce no cross-namespace traffic

## AWS Managed Services

| Service | AWS Replacement |
|---------|----------------|
| Self-managed PostgreSQL | AWS RDS (PostgreSQL, Multi-AZ for prod) |
| Self-managed Redis | AWS ElastiCache (Redis cluster mode) |
| Cloudinary | AWS S3 + CloudFront CDN |
| Self-managed Kafka | Amazon MSK |

**New environment variables:**
```
AWS_REGION
AWS_S3_BUCKET_NAME
AWS_CLOUDFRONT_DOMAIN
RDS_ENDPOINT
ELASTICACHE_ENDPOINT
MSK_BOOTSTRAP_SERVERS
```

---

## Multi-Tenant Network Isolation

- Each agency account's services run in isolated Kubernetes namespaces
- Network policies block cross-tenant traffic
- AWS IAM roles per service (no shared credentials)
- VPC with private subnets for all data stores

---

## Grafana + Prometheus

- Prometheus scrapes all services (Spring Boot Actuator `/actuator/prometheus` endpoints)
- Grafana dashboards: API health, scheduler outcomes, Kafka lag, RDS metrics, cache hit rate
- PagerDuty or Opsgenie integration for alert routing

**Alert upgrades from V1:**
- Kafka consumer lag > threshold → publisher worker auto-scaled
- RDS connection pool near limit → alert + auto-scaling trigger
- HPA scale events logged for capacity planning

---

## Backup and Recovery Upgrades

- RDS automated backups: daily snapshots, 30-day retention in prod
- Point-in-time recovery (PITR) enabled for prod RDS
- S3 versioning enabled for image bucket
- RPO target upgrade: 1 hour (from 24 hours in V1)
- RTO target upgrade: 2 hours (from same-day in V1)

## Runbook Additions

### Kubernetes Pod Crash Loop
1. `kubectl logs {pod} -n autoposter-prod` to get crash reason
2. Check resource limits (`kubectl describe pod {pod}`)
3. Check liveness probe configuration
4. Roll back deployment if crash is introduced by new release

### RDS Failover
1. AWS RDS Multi-AZ handles automatic failover (< 60 seconds)
2. Verify application reconnects after failover (connection pool retry)
3. Alert if failover exceeds 2 minutes

### HPA Scaling Event Investigation
1. Check HPA events: `kubectl describe hpa -n autoposter-prod`
2. Review Prometheus metrics that triggered scale
3. If unexpected: check for traffic spike or runaway job
